# 项目总策划案

## 一、项目定位

### 1.1 项目名称
暂定名：类《暗黑地牢》2D 回合制 RPG 项目

### 1.2 产品目标
本项目从第一天起即以“最终完整商业成品”为目标进行架构设计，而非以粗糙 MVP 为目标。开发节奏上采用分阶段拆解、逐步实现的方式，但每一个阶段的设计都必须服务于最终成品，不允许通过临时性、一次性、难以维护的方案堆砌功能。

### 1.3 核心体验
- 局内：高压、高风险、强策略性的地牢探索与回合制战斗
- 局外：围绕英雄名册、设施投资、资源分配的长期经营与恢复循环
- 整体：角色并非绝对服从的棋子，而是会承受压力、产生缺陷、形成长期成长与创伤的“人”

### 1.4 对标认知
本项目在系统复杂度与设计哲学上参考《暗黑地牢》，但在实际实现上采用 Godot 4.x、GDScript 2.0、数据驱动与状态机分层的现代工程方式进行重构，不照搬其历史实现形式。

## 二、开发总原则

### 2.1 双域严格拆分
整个项目在宏观上严格拆分为两个核心域：

- `城镇域`
  负责局外运营、英雄名册、设施、资源池、编队、远征结算回写
- `地牢域`
  负责局内探索、光照、遭遇、战斗、压力、掉落、撤退、任务推进

两个域从目录、场景、脚本职责、数据边界上都必须明确分离，不能混写。

### 2.2 数据驱动与状态机并重
- 局外系统以 `界面 + 数据驱动 + 持久化修改` 为核心
- 局内系统以 `状态机 + 规则解析 + 逻辑驱动` 为核心
- 复杂机制尽量通过 `资源定义 + 运行时状态 + 拦截链 / 触发器` 扩展，而不是写死在 UI 或单个脚本里

### 2.3 以可扩展为第一优先级
底层数据结构必须从第一版开始为以下机制预留扩展接口：
- 压力
- 折磨与美德
- 怪癖
- 疾病
- Buff / Debuff
- DOT / HOT
- 饰品
- 光照修正
- 死亡之门
- 尸体位
- 被动触发器

### 2.4 美术占位与正式美术兼容
前期将使用 AI 生成的正方形纯色背景图作为 Placeholder。后期将无缝替换为正式像素美术。因此：

- 不允许把尺寸写死到 UI 布局逻辑中
- 不允许让贴图原始尺寸决定交互规则
- `Control` 界面优先使用容器、自适应布局、相对锚点、动态缩放
- 战斗站位逻辑只认“逻辑槽位”，不认屏幕像素位置

### 2.5 工程规范
- 代码与 API 使用 Godot 4.x 语法标准
- 涉及操作说明、快捷键、路径示例时，以 windows 习惯为准
- 在给出具体代码前，先给出实现思路、节点树、数据结构或伪代码

## 四、核心产品结构

### 4.1 总体结构
项目整体由三层主流程构成：

- 应用层
  负责启动、菜单、档案载入、状态切换、全局服务
- 城镇域
  负责局外运营与长期状态管理
- 地牢域
  负责一次远征中的探索、战斗与会话状态

主流程如下：

`启动 -> 主菜单 -> 战役壳层 -> 城镇 -> 出征准备 -> 地牢探索 -> 战斗 / 事件 -> 远征结算 -> 返回城镇`

### 4.2 战役壳层
战役壳层是项目的总流程容器，负责：
- 持有当前 `战役状态`
- 在城镇域与地牢域之间切换
- 接收 `远征结果` 并回写
- 生成下一次 `出征准备`

城镇域与地牢域并列挂载，不彼此嵌套。

## 五、核心机制拆解

### 5.1 1-4 号位系统
战斗双方各拥有 4 个逻辑战斗位。技能的发动资格与目标合法性严格由站位决定。

每个技能至少包含三类约束：
- 施法者可用站位
- 目标可选站位
- 目标阵营与目标规则

这一系统的核心价值在于：
- 让职业设计、技能设计、敌人设计共享同一套空间规则
- 让位移、击退、拉拽、换位具备战略价值
- 让“错误站位”直接影响技能可用性与目标选择

项目实现上，战斗规则只认逻辑槽位，不依赖屏幕坐标。

### 5.2 回合排程
战斗回合采用“每轮重新生成行动顺序”的离散模型。

核心流程：
1. 每轮开始收集所有可行动单位
2. 计算行动值
3. 按行动值排序
4. 依次行动
5. 全员行动后开启下一轮

推荐公式：

`行动值 = 当前速度 + 随机浮动 + 环境 / 状态修正`

这样既能体现角色速度倾向，也能保留不确定性与状态修正空间。

### 5.3 压力与精神状态机
压力值范围为 `0 - 200`，不是第二条生命，而是控制权污染系统。

关键阶段：
- `0 - 99`：正常累积阶段
- `100`：触发精神检定
- `100 - 199`：高压阶段
- `200`：极限崩溃阶段

精神状态不只是数值惩罚，而是通过专门的拦截链介入行动流程：

`行动意图 -> 精神状态拦截 -> 普通状态拦截 -> 最终执行`

折磨会导致：
- 拒绝命令
- 强制跳过
- 改写目标
- 额外制造队友压力
- 插入自发行为或台词事件

### 5.4 光照系统
亮度范围为 `0 - 100`，本质上是一次远征中的全局环境修正器。

它应动态影响：
- 战斗难度
- 遭遇倾向
- 侦查与陷阱
- 掉落收益
- 压力环境

实现上，光照通过 `光照控制器 -> 光照区间定义 -> 当前地牢修正` 的形式提供给探索、战斗、掉落等系统读取。

### 5.5 局外经济循环
局外经济闭环如下：

`远征带回资源 -> 投入设施与英雄 -> 英雄更强或更稳定 -> 下一次远征更高效 -> 带回更多资源`

同时存在负向闭环：

`高压 / 疾病 / 怪癖 / 死亡 -> 恢复和补员成本提升 -> 减少成长投入 -> 下次远征更难`

这一闭环是城镇域存在的核心意义。### 5.6 英雄技能系统
每个英雄固定拥有 `4` 个可装备技能位。

规则如下：
- 第 `1` 个技能位固定为该英雄的专属技能
- 另外 `3` 个技能位通过“技能书”学习后，从已学技能池中自由搭配
- 出征前必须明确当前装备的 `4` 个技能组合

技能系统在数据层上应明确区分：
- `英雄定义`
  负责定义该英雄的专属技能
- `英雄战役状态`
  负责记录该英雄已经学会了哪些技能
- `出征准备数据`
  负责记录这次远征实际装备了哪 4 个技能

技能书系统的定位如下：
- 技能书是局外成长资源
- 技能书用于让指定英雄永久学会一个新技能
- 学会后的技能进入该英雄的“已学技能池”
- 出征前再从已学技能池中选择要装备的 3 个技能

### 5.7 英雄装备系统
每个英雄固定拥有 `4` 个装备位：

- `武器`
- `衣服`
- `饰品`
- `饰品`

装备槽位有严格类型限制：
- 武器只能装备在武器槽
- 衣服只能装备在衣服槽
- 饰品只能装备在两个饰品槽位

装备设计整体参考《暗黑破坏神 2》的思路，采用“基础装备 + 随机词条”的结构，而非固定数值装备。

### 5.8 装备品质与词条
装备品质分为四级：

- `绿色`
- `蓝色`
- `紫色`
- `橙色`

品质与随机词条数量强绑定：
- 绿色词条数量最少
- 蓝色词条数量较少
- 紫色词条数量较多
- 橙色词条数量最多

装备系统必须支持：
- 同名装备掉落出不同词条组合
- 同品质装备出现不同随机数值
- 装备实例化生成，而不是只依赖静态模板

因此装备数据结构必须区分：
- `装备定义`
  描述装备基础类型、槽位、基础属性范围、词条池、品质规则
- `装备实例`
  描述某一件具体装备随机到了哪些词条与具体数值
- `词条定义`
  描述一个词条会影响什么属性、作用于什么系统、数值范围如何生成

## 七、城镇域设计

### 7.1 城镇域职责
城镇域负责：
- 英雄名册管理
- 招募
- 设施服务
- 资源池管理
- 技能书学习
- 装备整理与穿戴
- 出征编队
- 远征结算回写
- 长期成长与恢复

### 7.2 城镇域的本质
城镇域本质上是：

`界面驱动 + 数据驱动 + 持久化修改`

它不直接处理战斗单位，不直接参与单场战斗规则解析。

### 7.3 名册系统
名册系统负责：
- 当前英雄列表
- 名册容量
- 招募候选池
- 英雄可用状态
- 死亡记录
- 设施占用状态
- 出征编队占用状态

同时，英雄战役状态还需要承载：
- 已学技能池
- 当前装备的 4 个技能位配置
- 当前装备栏中的武器、衣服、饰品、饰品
- 长期持有或穿戴的装备实例引用

### 7.4 设施系统
所有设施统一采用如下模型：

`设施定义 + 服务定义 + 设施运行状态`

第一批核心设施：
- 驿站：招募与名册扩展
- 铁匠铺：武器与护甲成长
- 行会：技能训练
- 减压设施：酒馆 / 修道院等
- 疗养院：疾病与怪癖处理

### 7.5 出征准备
出征准备负责：
- 选择最多 4 名英雄
- 调整队伍顺序
- 配置技能装配
- 检查专属技能与已学技能的合法组合
- 配置武器、衣服、饰品、饰品
- 准备补给
- 生成 `出征准备数据`

## 九、战斗系统设计

### 9.1 战斗系统职责
战斗系统负责：
- 1-4 号位管理
- 回合顺序
- 技能合法性判定
- 行动解析
- 伤害与治疗
- 状态效果
- 位移
- 死亡与尸体位
- 敌人 AI

### 9.2 战斗核心模块
推荐拆分为：
- 编队服务
- 回合顺序服务
- 属性服务
- 状态控制器
- 压力控制器
- 精神状态控制器
- 被动触发控制器
- 行动解析器
- 效果执行器
- 伤害解析器
- 位移解析器
- 死亡解析器
- 尸体控制器
- AI 意图构建器

### 9.3 行动链
统一行动链如下：

`行动意图 -> 合法性校验 -> 精神状态拦截 -> 普通状态拦截 -> 被动预处理 -> 生成最终行动 -> 执行效果 -> 死亡与阵型更新 -> 被动后处理`

### 9.4 复杂扩展机制
战斗系统需要支持：
- Buff / Debuff
- DOT / HOT
- 标记
- 守护
- 反击
- 尸体位
- 死亡之门
- 被动触发器
- 光照修正
- 怪癖 / 疾病 / 饰品投影
- 装备随机词条投影

## 十、表现与美术管线

### 10.1 占位图策略
前期使用纯色 Placeholder 资源，只负责流程跑通，不承载最终视觉判断。

### 10.2 正式美术替换原则
后期替换像素美术时，只允许主要替换：
- 贴图
- 主题
- 动画
- 音效资源
- 局部展示参数

不允许因为替换素材而重写以下内容：
- UI 布局结构
- 站位逻辑
- 交互逻辑
- 核心节点树

### 10.3 UI 布局原则
- 所有 `Control` 界面优先使用容器
- 顶栏、侧栏、底栏、弹窗统一采用自适应布局
- 英雄卡片、技能条、设施面板、日志面板采用可复用组件设计

### 10.4 表现层与逻辑层边界
逻辑层通过结构化事件通知表现层：
- 播放动画
- 更新日志
- 弹出伤害或压力反馈
- 播放音效

逻辑层不直接控制动画细节。

## 十四、当前结论

本项目当前的整体规划已经明确为：

- 以双域架构为基础的生产级 2D 回合制 RPG
- 局外采用城镇运营闭环
- 局内采用探索 + 战斗 + 光照 + 压力的状态机驱动结构
- 全项目采用中文目录与中文文件命名
- 前期 Placeholder 与后期像素美术可无缝替换

本策划案将作为后续工程落地、系统细化与内容制作的总纲。





## 热更新

- 需要更新行动节点计算逻辑，应该是跑圈逻辑，设定一个目标值，速度越快越先跑到终点就可以行动，然后继续跑，每次都是跑到终点然后行动，而不是固定回合每个回合内算速度
- 英雄除了技能之外，还应该有天赋系统
- 属性系统需要细化，英雄分为力量、敏捷、智力英雄，每个英雄不同的属性都有固定的成长值（影响每升一级增加的属性多少），力量（影响血量和物理攻击力），敏捷（影响护甲和速度），智力（影响魔法值和魔法抗性）
- 怪物需要增加词缀系统，精英怪物会附加各种词缀来强化，如物理免疫，火免疫，冰免疫，法力燃烧，力量强化等等
