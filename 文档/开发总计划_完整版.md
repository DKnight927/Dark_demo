# 黑暗之源开发总计划完整版

这份文件的目标不是解释单个系统，而是给后续开发一个统一总控面板。  
阅读它之后，开发者或 agent 应该能回答四个问题：

1. 现在项目处在哪个阶段
2. 下一步应该先做什么
3. 每个模块的完成标准是什么
4. 多个 agent 同时推进时怎么避免互相打架

---

## 一、项目当前状态判断

当前仓库已经具备三层基础：

- 策划层：
  拆分文档、规则源头、单元模板、协作元信息已经建立。
- 结构层：
  `文档 / 数据 / 脚本 / 场景` 的正式目录骨架已经建立。
- 起步实现层：
  战斗域已有原型脚本，且第一批状态效果与怪物词缀资源已经落地。

当前仍未完成的关键部分：

- 正式资源层远未补齐
- 城镇域 / 地牢域 / 应用流程还只是骨架
- 首发英雄与第一章敌人大多仍是“摘要内容”
- 主流程场景还没有串起来

结论：
项目现在不再是“前策划阶段”，而是“正式预制作阶段”。  
最合理的目标不是盲目扩内容，而是尽快跑通“第一章最小可玩闭环”。

---

## 二、总目标

总目标分三层：

### G1. 预制作完成

达到标准：

- 所有正式目录可承载开发
- 规则源头清晰，不再依赖大文档口头说明
- 首批资源类型脚本齐备
- 第一章最小可玩所需文档、任务、字段映射、资源目录齐备

### G2. 第一章最小可玩完成

达到标准：

- 可从主流程进入战役
- 可从城镇编队进入第一章地牢
- 可触发战斗、事件、掉落和返城
- 可击败第一章 Boss
- 可保存并读取基础战役状态

### G3. 内容扩展稳定化

达到标准：

- 新增英雄、怪物、词缀、事件时不需要重写底层
- 多 agent 可以并行扩内容，不互相破坏规则源头

---

## 三、唯一主线目标

当前唯一主线目标应固定为：

**跑通第一章最小可玩闭环。**

这意味着现阶段所有任务都要服务于下面这个流程：

`启动 -> 主菜单 -> 战役壳层 -> 城镇 -> 组队 -> 进入第一章地牢 -> 战斗 / 事件 -> Boss -> 结算 -> 返回城镇 -> 存档`

凡是不直接服务这条主线的工作，都应降优先级。

---

## 四、阶段划分

### Phase A：资源基础层完成

目标：

- 把文档中的最小可玩资源转成正式 `.tres`

范围：

- 状态效果
- 怪物词缀
- 敌方基础技能
- 首发 4 英雄技能
- 首发 4 英雄定义
- 第一章敌人定义
- 第一章 Boss 定义

完成标准：

- 不再依赖演示 JSON 或演示工厂也能组出一场战斗

### Phase B：战斗资源化完成

目标：

- 让战斗调试场景完全使用正式资源

范围：

- 调试入口从硬编码改为读资源
- 技能、状态、词缀、敌人、英雄全部走正式定义

完成标准：

- 战斗控制器可从资源直接构建一场战斗

### Phase C：首发内容定稿

目标：

- 把首发 4 英雄与第一章怪物补齐到正式生产规格

范围：

- 剑圣、盾卫、弓手、法师
- 腐化民兵、腐化祭司、腐化骑士长、腐化领主
- 第一章事件
- 第一章地牢节点配置

完成标准：

- 这些条目都不再标记为“摘要，待扩展”

### Phase D：长期状态系统接入

目标：

- 把特质、怪癖、疾病从文档推进到运行时可保存状态

范围：

- 资源定义
- 招募自带
- 远征结束获取
- 事件结果引用
- 城镇治疗 / 清除接口预留

完成标准：

- 战役状态可稳定保存这些长期条目

### Phase E：主流程骨架完成

目标：

- 建立完整应用流程壳层

范围：

- 应用根
- 主菜单
- 战役流程控制器
- 城镇控制器
- 地牢控制器
- 存档管理器

完成标准：

- 项目具备主流程最小切换能力

### Phase F：第一章闭环联调

目标：

- 串通完整最小可玩循环

范围：

- 城镇出征
- 地牢探索
- 战斗
- 事件
- Boss
- 结算
- 返回城镇

完成标准：

- 一轮完整远征可实际跑完

---

## 五、当前固定优先级

优先级从高到低如下：

1. 正式资源化
2. 战斗从演示数据切到正式资源
3. 首发内容定稿
4. 主流程壳层
5. 长期状态系统接入
6. UI 细化
7. 第二章及之后的内容扩充

明确不要现在优先做的事：

- 扩更多职业
- 扩更多怪物
- 扩更多地牢
- 大规模做 UI 美术
- 过早做复杂优化

## 六、模块任务总表

### M1. 状态效果资源

必须完成：

- 流血
- 中毒
- 燃烧
- 霜冻
- 眩晕
- 虚弱
- 缓速
- 束缚
- 暴露
- 动摇
- 士气低落
- 士气高昂
- 铁壁
- 腐化创伤
- 绝望
- 剧毒
- 魔力封印

验收标准：

- 可被技能效果正确引用
- 可被运行时状态实例化
- 持续回合和层数规则有资源字段承载

### M2. 怪物词缀资源

必须完成：

- 第一批 8 个基础词缀
- 第二批中层词缀
- 第三批深层词缀

验收标准：

- 可被敌人定义引用
- 精英化时可以按池抽取
- 公共规则仍以词缀系统正文为准

### M3. 敌方技能资源

必须完成：

- 第一章怪物和 Boss 的全部技能

验收标准：

- 每个敌方技能都能被敌人资源引用
- 技能效果类型与状态效果引用正确

### M4. 英雄技能资源

必须完成：

- 首发 4 英雄的专属技能 1-4 级
- 首发 4 英雄的传承技能 1-4 级

验收标准：

- 能用正式技能资源替换演示技能

### M5. 英雄定义资源

必须完成：

- 剑圣
- 盾卫
- 弓手
- 法师

验收标准：

- 基础属性、成长、技能引用、天赋引用、推荐站位完整

### M6. 敌人定义资源

必须完成：

- 腐化民兵
- 腐化祭司
- 腐化骑士长
- 腐化领主

验收标准：

- 属性、技能、AI、词缀池、掉落完整

### M7. 特质 / 怪癖 / 疾病资源

必须完成：

- 特质至少 12 条
- 怪癖至少 20 条
- 疾病至少 8 条

验收标准：

- 可被战役状态记录
- 可被事件和远征结果引用

### M8. 主流程脚本骨架

必须完成：

- 应用根
- 战役流程控制器
- 城镇控制器
- 地牢控制器
- 存档管理器

验收标准：

- 脚本可实例化
- 主流程切换不再全依赖调试入口

### M9. 正式场景骨架

必须完成：

- 应用根场景
- 主菜单场景
- 城镇壳层场景
- 地牢壳层场景

验收标准：

- 场景路径稳定
- 流程控制器可以挂接

## 七、并行开发规则

为了方便多个 agent 并行，强制按下面方式拆分：

### 线程 A：资源组

负责：

- `数据/定义/状态效果`
- `数据/定义/怪物词缀`
- `数据/定义/技能`

不要改：

- 主流程脚本
- 城镇域 / 地牢域脚本

### 线程 B：英雄与敌人组

负责：

- `数据/定义/英雄`
- `数据/定义/敌人`
- 英雄与怪物文档定稿

不要改：

- 公共战斗规则源头

### 线程 C：流程组

负责：

- `脚本/应用`
- `脚本/应用流程`
- `脚本/领域/城镇域`
- `脚本/领域/地牢域`
- `脚本/领域/共用`
- 正式场景壳层

不要改：

- 英雄个体和怪物个体资源内容

### 线程 D：长期状态组

负责：

- 特质 / 怪癖 / 疾病资源
- 招募与远征结果接口

不要改：

- 战斗控制器核心流程

---

## 八、协作硬规则

1. 系统规则只能改系统目录，不能把公共规则写回英雄或怪物单文件
2. 英雄和怪物文件只负责个体内容
3. 文档和资源冲突时，先修规则源头，再修资源
4. 资源命名必须遵守现有命名清单
5. 若新增目录，必须补 `README`
6. 若新增系统级规则，必须更新规则源头说明

---

## 九、定义完成的标准

一个模块只有满足下面标准，才算“完成”，不能只做到一半就宣布结束：

### 文档完成

- 有规则源头
- 有字段说明
- 有目录入口
- 有冲突处理说明

### 脚本完成

- 能实例化
- 结构不与现有命名冲突
- 有稳定入口路径

### 资源完成

- 有正式目录
- 有正式命名
- 能被脚本引用
- 能在 Godot 中被加载

### 流程完成

- 至少能走一遍最小链路
- 不是只在调试入口里单步验证

## 十、推荐立即执行方案

如果要从现在开始连续推进，不再中断，建议直接按下面顺序执行：

1. 补全剩余状态效果资源
2. 补全剩余怪物词缀资源
3. 建第一章敌方技能资源
4. 建首发 4 英雄技能资源
5. 建首发 4 英雄定义资源
6. 建第一章敌人和 Boss 定义资源
7. 建特质 / 怪癖 / 疾病资源
8. 建主流程与域控制器场景壳层
9. 让战斗调试入口切换到正式资源
10. 跑通第一章最小可玩闭环

## 十一、当前一句话判断

**后续最重要的事已经不是继续写策划，而是把现有策划稳定转成正式资源和正式流程。**
